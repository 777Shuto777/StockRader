## 2. システム概要

本システムは、日本株の個別銘柄に関する適時開示情報、材料キーワード、監視リスト、取引計画、検証ログを一元管理するための個人用ダッシュボードである。

目的は、急騰候補銘柄を自動で売買判断することではなく、投資判断前の候補抽出・材料整理・リスク管理・振り返りを効率化することである。

v0.1.3では、単なる材料記録ツールではなく、**出口のないエントリーを防ぐこと**を重要な設計方針に加える。

---

## 3. 開発目的

現在、個別株の材料確認には複数の情報源を見る必要がある。

* 適時開示情報
* 値上がり率ランキング
* ストップ高銘柄一覧
* 出来高急増ランキング
* 決算発表予定
* 業績修正リリース
* 株式ニュース

これらを毎日個別に確認するのは手間がかかり、継続しにくい。

また、材料株の短期売買では以下の問題が起きやすい。

* 勢いで買う
* 出口を決めずに買う
* 損切りを先送りする
* サイズを感覚で決める
* 見送った判断を振り返らない
* 後から都合よく判断理由を書き換える

本システムでは、まずMVPとして「適時開示情報を起点に、気になる銘柄を発見し、監視リストに入れ、買う場合は出口と許容損失を事前に固定し、後から検証する」流れを作る。

## 4. システムの基本方針

### 4.1 判断代行ではなく判断補助

本システムは、銘柄の買い・売りを自動判定しない。

あくまで以下を支援する。

* 材料の早期発見
* 材料の分類
* 気になる銘柄の保存
* 取引計画の作成
* 出口条件の事前固定
* 許容損失からの株数逆算
* 判断理由の記録
* 結果の振り返り

### 4.2 MVPでは機能を絞る

v0.1.3では、以下に集中する。

* 適時開示の一覧管理
* キーワードによる材料分類
* 監視リスト登録
* 銘柄メモ
* 取引計画
* 出口の事前固定
* リスク逆算サイジング
* 検証ログ
* Vercel + Supabase でのクラウド利用

価格データ、出来高データ、AI要約、通知、自動取得、自動スコアリングの高度化は後続バージョンで検討する。

### 4.3 継続利用を最優先する

本システムの成功条件は、機能の多さではなく、毎日短時間で使えることである。

v0.1.3の目標は、毎日5分で「今日見るべき材料銘柄」と「取引してよい計画か」を確認できる状態を作ることである。

### 4.4 出口なき買いを防ぐ

v0.1.3では、買い判断を記録する場合、以下を必須とする。

* エントリー予定価格
* 損切価格
* 利確目標価格
* 最大保有日数
* 口座資産スナップショット
* 1取引あたりの許容リスク%
* 許容損失額
* 推奨株数

これらが未入力の場合、買い計画を確定できない。

ただし、下書き状態では保存可能とする。
下書きは「検討メモ」であり、「実行可能な買い計画」とは扱わない。

### 4.5 サイズを先に決めない

株数は感覚で決めない。

株数は以下の順で決定する。

1. 口座資産を入力する
2. 1取引あたりの許容リスク%を入力する
3. エントリー予定価格を入力する
4. 損切価格を入力する
5. 許容損失額を算出する
6. 1株あたり損失額を算出する
7. 推奨株数を算出する

先に「何株買いたいか」を決めるのではなく、損切りになった場合に許容できる損失額から株数を逆算する。

### 4.6 事前計画を後から書き換えない

買い計画を確定した後は、以下を原則編集不可とする。

* 仮説
* エントリー予定価格
* 損切価格
* 利確目標価格
* 最大保有日数
* 許容リスク%
* 推奨株数

結果、学び、実績値は後から追記できる。

入力ミスなどで修正が必要な場合は、後続バージョンで変更履歴を残す方式を検討する。
v0.1.3では、まず「確定後は主要な事前計画を上書きしない」方針を優先する。

* 無料枠を活用してMVPを構築したい

## 6. 解決したい課題

### 6.1 情報収集が分散している

複数サイトを確認する必要があり、毎日の継続が難しい。

### 6.2 材料を見つけても記録が残らない

気になった銘柄を見つけても、なぜ気になったのか、買ったのか、見送ったのかが残りにくい。

### 6.3 振り返りができない

翌日や1週間後に、その判断が正しかったのか検証しにくい。

### 6.4 上がった理由を後追いでしか学べない

ストップ高後に理由を調べるだけでは、発見力が育ちにくい。

### 6.5 ローカル環境だけでは継続しにくい

ローカルPC上だけで動かす場合、外出先や別端末から確認しにくい。

v0.1.3では、最初からVercel + Supabaseを利用し、ブラウザからアクセスできる状態を目指す。

### 6.6 出口を決めないまま買ってしまう

材料株では、上昇期待だけで買い、下がった時の対応を決めていないことが大きな損失につながる。

v0.1.3では、買い計画を確定する前に損切価格・利確目標・最大保有日数を必須化する。

### 6.7 株数を感覚で決めてしまう

「買える金額」や「なんとなくの株数」で入ると、損切時の損失額が大きくなりやすい。

v0.1.3では、許容損失から推奨株数を逆算する。

### 6.8 後から都合よく判断理由を書き換えてしまう

取引後に「最初からこう考えていた」と都合よく記録を書き換えると、検証が機能しない。

v0.1.3では、取引計画を確定した後は、主要な事前計画をロックする。

## 8. 主要ユースケース

### 8.1 当日の注目開示を登録する

ユーザーはTDnet等で気になる開示を見つけたら、自作アプリに手動登録する。

登録対象は全開示ではなく、ユーザーが気になった開示に限定する。

登録項目例：

* 開示日時
* 銘柄コード
* 銘柄名
* 開示タイトル
* 開示URL
* メモ

### 8.2 強材料候補を見つける

システムは開示タイトルに含まれるキーワードをもとに、材料カテゴリを自動分類する。

例：

* 上方修正
* 黒字転換
* 大口受注
* 自社株買い
* 増配
* TOB
* MBO
* 業務提携
* 資本提携
* 希薄化
* 継続企業リスク
* 業績修正要確認

### 8.3 気になる銘柄を監視リストに追加する

ユーザーは、気になる開示や銘柄を監視リストに登録する。

登録時に、以下を記録できる。

* 気になった理由
* 材料の印象
* 後で確認したいポイント
* 監視メモ

### 8.4 買い計画を作成する

ユーザーが買いを検討する場合、取引計画を作成する。

買い計画では以下を必須入力とする。

* 仮説
* エントリー予定価格
* 損切価格
* 利確目標価格
* 最大保有日数
* 口座資産スナップショット
* 1取引あたりの許容リスク%
* 実際に買う予定株数

システムは、許容損失額から推奨株数を自動計算する。

### 8.5 見送り計画を作成する

ユーザーは、買わない判断をした銘柄も記録できる。

見送り計画では以下を記録する。

* 見送った理由
* 買わない条件
* 後で確認する日
* 想定シナリオ

見送り判断の詳細な採点はv0.2以降で実装する。

### 8.6 取引計画を確定する

ユーザーは、入力内容を確認したうえで取引計画を確定する。

確定後は、以下を編集不可とする。

* 仮説
* エントリー予定価格
* 損切価格
* 利確目標価格
* 最大保有日数
* 許容リスク%
* 推奨株数

### 8.7 後日、判断結果を振り返る

翌日、3日後、1週間後などに、当時の判断が正しかったかを振り返る。

検証ログには以下を記録する。

* 結果
* 学び
* 計画通りだったか
* 計画とズレた点
* 次回ルール

### 8.8 外出先や別端末から確認する

ユーザーは、Vercelにデプロイされたアプリへブラウザからアクセスし、登録済みデータや監視リストを確認する。

## 9. 画面要件

### 9.1 画面構成

v0.1.3では、Next.jsのルーティングを前提に以下の画面を作成する。

| ルート            | 画面名     | 目的                  |
| -------------- | ------- | ------------------- |

| `/watchlist`   | 監視リスト画面 | 気になる銘柄を管理する         |
| `/trade-plans` | 取引計画画面  | 買い・見送り・監視継続の計画を管理する |

* Card

* Input

* Select
* Badge

* Tabs

* Form validation message

### 9.2 注目開示画面

#### 目的

登録した注目開示を一覧で確認する。

| 項目      | 内容               |
| ------- | ---------------- |

| 確認済み    | ユーザーが確認したか       |
| 監視追加    | 監視リストへの追加ボタン     |
| 取引計画作成  | 取引計画への遷移ボタン      |

#### 主な操作

* 開示一覧を見る
* 注目開示を手動登録する
* 材料カテゴリで絞り込む
* 銘柄コードで検索する
* 監視リストへ追加する
* 取引計画を作成する
* 確認済みにする

### 9.3 監視リスト画面

#### 目的

気になる銘柄を保存し、継続的に確認する。

#### 表示項目

| 項目     | 内容                  |
| ------ | ------------------- |
| 登録日    | 監視リストに入れた日          |
| 銘柄コード  | 銘柄コード               |
| 銘柄名    | 銘柄名                 |
| 監視理由   | なぜ気になったか            |
| 材料カテゴリ | 大口受注、上方修正など         |
| ステータス  | 未確認、調査中、買い検討、見送り、終了 |
| 監視メモ   | その監視時点の仮説・注意点       |
| 関連開示   | 元になった開示情報           |
| 取引計画   | 関連する取引計画            |

#### 銘柄メモと監視メモの違い

| 種類   | 保存先   | 役割                     |
| ---- | ----- | ---------------------- |
| 銘柄メモ | 銘柄マスタ | 銘柄全体に関する継続的なメモ         |
| 監視メモ | 監視リスト | その材料・そのタイミングに対する一時的なメモ |

#### 主な操作

* 監視銘柄を追加する
* ステータスを変更する
* 監視メモを編集する
* 取引計画を作成する
* 監視終了にする

---

### 9.4 取引計画画面

#### 目的

買い・見送り・監視継続の判断を事前に記録する。

#### 計画種別

| 種別    | 内容              |
| ----- | --------------- |
| 買い計画  | 実際に買う可能性がある計画   |
| 見送り計画 | 買わない判断を記録する計画   |
| 監視継続  | 今は判断せず、継続確認する計画 |

#### 買い計画の表示項目

| 項目           | 内容                 |
| ------------ | ------------------ |
| 計画日          | 計画を作成した日           |
| 銘柄コード        | 銘柄コード              |
| 銘柄名          | 銘柄名                |
| 仮説           | なぜ上がると思うのか         |
| エントリー予定価格    | 入る予定の価格            |
| 損切価格         | 損切りする価格            |
| 利確目標価格       | 利益確定を検討する価格        |
| 最大保有日数       | 何日以内に決着をつけるか       |
| 口座資産スナップショット | 計画時点の口座資産          |
| 許容リスク%       | 1取引あたり何%まで損失を許容するか |
| 許容損失額        | 口座資産 × 許容リスク%      |
| 1株あたり損失額     | エントリー予定価格 − 損切価格   |
| 推奨株数         | 許容損失額 ÷ 1株あたり損失額   |
| 実際予定株数       | 実際に買う予定の株数         |
| 計画ステータス      | 下書き、確定、実行済み、終了     |
| ロック状態        | 確定後に主要項目を編集不可にする   |

#### 見送り計画の表示項目

| 項目     | 内容            |
| ------ | ------------- |
| 計画日    | 計画を作成した日      |
| 銘柄コード  | 銘柄コード         |
| 銘柄名    | 銘柄名           |
| 見送り理由  | なぜ買わないのか      |
| 想定シナリオ | 上がる可能性・下がる可能性 |
| 再確認日   | 後で見直す日        |
| 関連開示   | 元になった開示情報     |

#### 主な操作

* 買い計画を作成する
* 見送り計画を作成する
* 監視継続計画を作成する
* 推奨株数を計算する
* 計画を確定する
* 確定後に主要項目をロックする
* 検証ログを作成する

### 9.5 検証ログ画面

#### 目的

売買判断や見送り判断を記録し、後から振り返る。

| 項目       | 内容                |

### 10.3 材料キーワード分類機能

#### 概要

開示タイトルに含まれるキーワードをもとに、材料カテゴリを自動分類する。

#### 判定対象

v0.1.3では、判定対象は開示タイトルのみとする。
本文PDFや詳細本文の解析は行わない。

#### マッチング方式

| 項目      | 方針                   |
| ------- | -------------------- |
| マッチ方式   | 部分一致                 |
| 大文字・小文字 | 区別しない                |
| 全角・半角   | 可能であれば正規化する          |
| 複数ヒット   | 全カテゴリを保持する           |
| 主カテゴリ   | 優先度が最も高いカテゴリを採用する    |
| 同優先度の場合 | 絶対値の大きいスコアを優先する      |
| 自動材料スコア | 主カテゴリのスコアを採用する。合算しない |

#### 初期キーワード

| カテゴリ     | キーワード例           | スコア | 優先度 |
| -------- | ---------------- | --: | --: |
| TOB      | 公開買付け、TOB        |  +6 | 100 |
| MBO      | MBO、マネジメント・バイアウト |  +6 | 100 |
| 上方修正     | 上方修正             |  +5 |  90 |
| 下方修正     | 下方修正             |  -5 |  90 |
| 黒字転換     | 黒字転換、営業利益黒字      |  +5 |  85 |
| 継続企業リスク  | 継続企業の前提、重要な疑義    |  -6 |  85 |
| 大口受注     | 大口受注、受注、契約締結     |  +4 |  80 |
| 自社株買い    | 自己株式取得、自社株買い     |  +4 |  75 |
| 増配       | 増配、配当予想の修正       |  +3 |  70 |
| 希薄化      | 新株予約権、MSワラント、希薄化 |  -4 |  70 |
| 業績修正要確認  | 業績予想の修正、通期業績予想   |   0 |  60 |
| 資本提携     | 資本提携             |  +2 |  55 |
| 業務提携     | 業務提携、協業、共同開発     |  +2 |  50 |
| 第三者割当要確認 | 第三者割当            |  -1 |  45 |

#### 注意点

* スコアは売買判断ではなく、調査優先度の目安として扱う
* 「業績予想の修正」は上方修正か下方修正かタイトルだけでは判断できない場合があるため、`業績修正要確認` とする
* 「第三者割当」は資本提携の場合も希薄化懸念の場合もあるため、v0.1.3では `第三者割当要確認` とする
* 本文解析を行うまでは、曖昧な材料を自動で強材料・悪材料に断定しない

### 10.7 リスク逆算サイジング機能

#### 概要

口座資産と許容リスク%から、推奨株数を自動計算する。

#### 計算式

```text
許容損失額 = 口座資産スナップショット × 許容リスク% ÷ 100
```

```text
1株あたり損失額 = エントリー予定価格 − 損切価格
```

```text
理論株数 = 許容損失額 ÷ 1株あたり損失額
```

```text
推奨株数 = 取引単位に合わせて切り下げた株数
```

#### 取引単位

日本株の通常売買を想定し、初期値は100株とする。
ただし、将来的な拡張に備え、設定値として変更可能な構造にする。

#### 推奨株数が0になる場合

推奨株数が取引単位未満の場合、システムは以下を表示する。

```text
この損切幅とリスク許容額では、最小単元を買うとリスク超過になります。
```

この場合、買い計画を確定できない。

#### サイズ超過

実際予定株数が推奨株数を超える場合、買い計画を確定できない。

v0.1.3では、サイズ超過を許可する例外入力は設けない。
例外許可や理由付きオーバーライドはv0.2以降で検討する。

## 12. データ要件

### 12.1 ID方針

各テーブルの `id` はCUIDを使用する。

理由：

* Prismaとの相性が良い
* PostgreSQLでも扱いやすい
* UUIDより短く、開発時に読みやすい

---

### 12.2 銘柄マスタ

#### 方針

銘柄マスタは銘柄の基本情報を保持する。
`code` はユニークとする。

未登録銘柄の開示情報が登録された場合は、銘柄マスタに自動登録する。

| カラム名       | 型        | 内容         |
| ---------- | -------- | ---------- |
| id         | string   | CUID       |
| code       | string   | 銘柄コード。ユニーク |
| name       | string   | 銘柄名        |
| market     | string   | 市場区分       |
| memo       | text     | 銘柄全体に関するメモ |
| created_at | datetime | 作成日時       |
| updated_at | datetime | 更新日時       |

---

### 12.3 開示情報

#### 方針

開示情報は `stock_code` で銘柄マスタと関連付ける。
ただし、開示時点の表示名を保持するため `stock_name_snapshot` を持つ。

| カラム名                | 型        | 内容                  |
| ------------------- | -------- | ------------------- |
| id                  | string   | CUID                |
| unique_key          | string   | 重複判定用キー。ユニーク        |
| disclosed_at        | datetime | 開示日時。UTC保存、JST表示    |
| stock_code          | string   | 銘柄コード。銘柄マスタのcodeと関連 |

## 19. 運用イメージ

### 19.1 朝

* 前日引け後から当日朝までの開示を確認する
* 気になる銘柄を注目開示として登録する
* 監視リストに追加する
* その日の確認対象を決める

### 19.2 昼

* 場中開示を確認する
* 必要に応じて注目開示や監視リストに追加する
* 買いを検討する場合は取引計画を作成する
* 出口・許容損失・推奨株数を確認する

### 19.3 エントリー前

* 仮説を記録する
* エントリー予定価格を入力する
* 損切価格を入力する
* 利確目標価格を入力する
* 最大保有日数を入力する
* 口座資産と許容リスク%を入力する
* 推奨株数を確認する
* 推奨株数を超えていないことを確認する
* 問題なければ取引計画を確定する

### 19.4 引け後

* 当日のストップ高や大きく動いた銘柄の理由を確認する
* 買った銘柄、見送った銘柄を検証ログに記録する
* 計画通りだったかを確認する
* 翌日見る銘柄を整理する

## 22. 注意事項

本システムは投資判断を補助する個人用ツールであり、利益を保証するものではない。

スコアやキーワード分類は、あくまで調査優先度を示すものであり、売買判断そのものではない。

推奨株数は、入力されたエントリー予定価格、損切価格、口座資産、許容リスク%をもとにした機械的な計算結果であり、利益を保証するものではない。

最終的な投資判断は、開示内容、業績、財務、需給、チャート、リスク許容度を確認したうえでユーザー自身が行う。

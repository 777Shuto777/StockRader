## 2. 基本方針

### 2.1 判断代行ではなく判断補助

本システムは、銘柄の売買判断を自動で行わない。

システムが行うのは以下である。

* 材料情報の整理
* 監視対象の記録
* 取引計画の記録
* リスク計算
* 出口条件の必須化
* 規律違反につながる入力のブロック
* 振り返りの記録

最終的な売買判断はユーザー自身が行う。

---

### 2.2 買い計画は「出口」がなければ確定不可

`plan_type = BUY` の取引計画を `COMMITTED` にする場合、以下が必須である。

* 仮説
* エントリー予定価格
* 損切価格
* 利確目標価格
* 最大保有日数
* 口座資産スナップショット
* 許容リスク%
* 許容損失額
* 1株あたり損失額
* 推奨株数
* 実際予定株数

これらが不足している場合、買い計画は確定できない。

---

### 2.3 株数は許容損失から逆算する

株数は、先に決めない。

以下の順で算出する。

1. 口座資産を入力する
2. 許容リスク%を入力する
3. エントリー予定価格を入力する
4. 損切価格を入力する
5. 許容損失額を算出する
6. 1株あたり損失額を算出する
7. 推奨株数を算出する
8. 実際予定株数が推奨株数以下か検証する

推奨株数を超える買い計画は確定できない。

---

### 2.4 確定済み計画は後から都合よく書き換えない

取引計画を `COMMITTED` にした後は、主要な事前計画を編集不可とする。

これは、後知恵で仮説や出口条件を書き換えることを防ぐためである。

## 3. 共通入力ルール

### 3.1 文字列共通

| ルール      | 内容                  |
| -------- | ------------------- |
| 前後空白     | 保存前にtrimする          |
| 空文字      | 任意項目では `null` として扱う |
| 必須項目の空文字 | エラー                 |
| 改行       | text項目では許容          |
| 最大長      | DB制約がある項目はDB定義に従う   |

---

### 3.2 銘柄コード

#### 対象

* `stocks.code`
* `disclosures.stock_code`
* `watchlist_items.stock_code`
* `trade_plans.stock_code`
* `verification_logs.stock_code`

#### ルール

| ルール  | 内容          |
| ---- | ----------- |
| 前後空白 | 除去する        |
| 英字   | 大文字に変換する    |
| 形式   | 4〜5文字の半角英数字 |
| 数字限定 | しない         |
| 不正形式 | エラー         |

#### 正常例

| 入力     | 保存値    |
| ------ | ------ |
| `6659` | `6659` |
| `7203` | `7203` |
| `abc1` | `ABC1` |
| `130A` | `130A` |

#### 異常例

| 入力       | 理由    |
| -------- | ----- |
| `123`    | 3文字   |
| `123456` | 6文字   |
| `６６５９`   | 全角    |
| `66-59`  | 記号を含む |
| 空文字      | 必須項目  |

---

### 3.3 日時

| 項目   | ルール                      |
| ---- | ------------------------ |
| 入力   | JSTとして受け取る               |
| 保存   | UTCで保存する                 |
| 表示   | JSTに変換して表示する             |
| 日付のみ | `review_date` などは日付として扱う |

---

### 3.4 数値

| 項目     | ルール      |
| ------ | -------- |
| 価格     | 0より大きい   |
| 金額     | 0以上      |
| 株数     | 0より大きい整数 |
| 許容リスク% | 0より大きい   |
| 最大保有日数 | 1以上の整数   |
| スコア    | 整数       |
| 優先度    | 整数       |

## 4. 注目開示登録ルール

### 4.1 対象テーブル

* `stocks`
* `disclosures`
* `keyword_rules`

---

### 4.2 入力項目

| 項目      |  必須 | 内容          |
| ------- | --: | ----------- |
| 開示日時    | Yes | JST入力       |
| 銘柄コード   | Yes | 4〜5文字の半角英数字 |
| 銘柄名     | Yes | 開示登録時点の銘柄名  |
| 開示タイトル  | Yes | 適時開示タイトル    |
| 開示URL   |  No | 元情報URL      |
| ユーザー重要度 |  No | 任意入力        |
| 確認済み    |  No | 初期値false    |

---

### 4.3 登録処理

注目開示登録時は、以下の順で処理する。

1. 入力値を正規化する
2. 銘柄コードを検証する
3. 銘柄マスタを確認する
4. 銘柄未登録の場合、`stocks` に作成する
5. `unique_key` を生成する
6. 重複開示がないか確認する
7. `keyword_rules` をもとに材料分類する
8. `disclosures` に保存する

---

### 4.4 銘柄マスタ自動登録ルール

`stocks` に対象銘柄コードが存在しない場合、自動作成する。

| カラム      | 値          |
| -------- | ---------- |
| `code`   | 正規化後の銘柄コード |
| `name`   | 入力された銘柄名   |
| `market` | null       |
| `memo`   | null       |

すでに同一コードが存在する場合、`stocks.name` は自動更新しない。
開示時点の銘柄名は `disclosures.stock_name_snapshot` に保存する。

---

### 4.5 重複判定ルール

`unique_key` は以下を正規化して生成する。

URLありの場合：

```text
disclosed_at + stock_code + title + url
```

URLなしの場合：

```text
disclosed_at + stock_code + title
```

生成した文字列をハッシュ化し、`disclosures.unique_key` に保存する。

同じ `unique_key` が存在する場合、登録エラーとする。

---

### 4.6 材料分類ルール

#### 判定対象

v0.1では、判定対象は `title` のみとする。

PDF本文、URL先本文、ニュース本文は判定対象外。

---

#### マッチング方式

| 項目        | ルール                                            |
| --------- | ---------------------------------------------- |
| マッチ方式     | 部分一致                                           |
| 大文字・小文字   | 区別しない                                          |
| 全角・半角     | 可能な範囲で正規化する                                    |
| 複数ヒット     | 全カテゴリを保持する                                     |
| 主カテゴリ     | 優先度が最も高いカテゴリ                                   |
| 同優先度の場合   | スコア絶対値が大きいカテゴリ                                 |
| さらに同順位の場合 | キーワード文字数が長いもの                                  |
| 自動材料スコア   | 主カテゴリのscore                                    |
| ヒットなし     | `primary_category = null`, `auto_score = null` |

---

#### matched_categories 保存形式

`matched_categories` には、以下のようなJSONを保存する。

```json
[
  {
    "category": "大口受注",
    "keyword": "大口受注",
    "score": 4,
    "priority": 80
  }
]
```

## 7. 買い計画確定ルール

### 7.1 対象条件

以下の条件を満たす場合に適用する。

```text
plan_type = BUY
かつ
plan_status を COMMITTED に変更する
```

---

### 7.2 必須項目

買い計画を確定する場合、以下をすべて必須とする。

| 項目           | カラム                         |
| ------------ | --------------------------- |
| 仮説           | `thesis`                    |
| エントリー予定価格    | `planned_entry_price`       |
| 損切価格         | `planned_stop_price`        |
| 利確目標価格       | `planned_take_profit_price` |
| 最大保有日数       | `max_holding_days`          |
| 口座資産スナップショット | `account_size_snapshot`     |
| 許容リスク%       | `risk_percent`              |
| 許容損失額        | `planned_risk_amount`       |
| 1株あたり損失額     | `planned_loss_per_share`    |
| 取引単位         | `trading_unit`              |
| 推奨株数         | `recommended_quantity`      |
| 実際予定株数       | `planned_quantity`          |

---

### 7.3 価格バリデーション

| ルール                  | エラー条件                                              |
| -------------------- | -------------------------------------------------- |
| エントリー予定価格は0より大きい     | `planned_entry_price <= 0`                         |
| 損切価格は0より大きい          | `planned_stop_price <= 0`                          |
| 利確目標価格は0より大きい        | `planned_take_profit_price <= 0`                   |
| 損切価格はエントリー予定価格より低い   | `planned_stop_price >= planned_entry_price`        |
| 利確目標価格はエントリー予定価格より高い | `planned_take_profit_price <= planned_entry_price` |

v0.1では買い方向のみを扱うため、空売りの価格関係は考慮しない。

---

### 7.4 最大保有日数バリデーション

| ルール        | エラー条件                  |
| ---------- | ---------------------- |
| 最大保有日数は1以上 | `max_holding_days < 1` |
| 整数のみ       | 小数や文字列                 |

---

### 7.5 リスク%バリデーション

| ルール           | エラー条件                                          |
| ------------- | ---------------------------------------------- |
| 許容リスク%は0より大きい | `risk_percent <= 0`                            |
| 許容リスク%は上限以下   | `risk_percent > app_settings.max_risk_percent` |

`max_risk_percent` が未設定の場合は、システムデフォルトとして `2` を使う。

---

### 7.6 取引単位バリデーション

| ルール      | エラー条件              |
| -------- | ------------------ |
| 取引単位は1以上 | `trading_unit < 1` |
| v0.1の初期値 | `100`              |

---

## 8. リスク逆算サイジングルール

### 8.1 計算タイミング

以下のタイミングでリスク計算を行う。

* 買い計画作成・更新時
* 買い計画確定時
* エントリー予定価格変更時
* 損切価格変更時
* 口座資産変更時
* 許容リスク%変更時
* 取引単位変更時

最終的には、確定時にサーバー側で必ず再計算する。

---

### 8.2 計算式

```text
planned_risk_amount = account_size_snapshot × risk_percent ÷ 100
```

```text
planned_loss_per_share = planned_entry_price − planned_stop_price
```

```text
raw_quantity = planned_risk_amount ÷ planned_loss_per_share
```

```text
recommended_quantity = floor(raw_quantity ÷ trading_unit) × trading_unit
```

---

### 8.3 計算例

| 項目        |          値 |
| --------- | ---------: |
| 口座資産      | 1,000,000円 |
| 許容リスク%    |         1% |
| エントリー予定価格 |       100円 |
| 損切価格      |        90円 |
| 取引単位      |       100株 |

計算：

```text
planned_risk_amount = 1,000,000 × 1 ÷ 100 = 10,000
```

```text
planned_loss_per_share = 100 − 90 = 10
```

```text
raw_quantity = 10,000 ÷ 10 = 1,000
```

```text
recommended_quantity = 1,000
```

---

### 8.4 推奨株数が0の場合

`recommended_quantity < trading_unit` の場合、推奨株数は `0` とする。

この場合、買い計画は確定不可。

表示メッセージ：

```text
この損切幅とリスク許容額では、最小単元を買うとリスク超過になります。
エントリー価格、損切価格、許容リスク%を見直してください。
```

---

### 8.5 サイズ超過ブロック

`planned_quantity > recommended_quantity` の場合、買い計画は確定不可。

表示メッセージ：

```text
予定株数が推奨株数を超えています。
株数を減らすか、損切価格・許容リスク%を見直してください。
```

---

### 8.6 planned_quantity の取引単位チェック

`planned_quantity % trading_unit != 0` の場合、買い計画は確定不可。

表示メッセージ：

```text
予定株数は取引単位の倍数で入力してください。
```

---

## 9. 見送り計画ルール

### 9.1 対象条件

```text

```

### 9.3 任意項目

以下は任意とする。

| 項目     | カラム             |
| ------ | --------------- |
| 想定シナリオ | `thesis`        |
| 関連開示   | `disclosure_id` |
| 関連監視   | `watchlist_id`  |

---

### 9.4 注意点

v0.1では、見送り後の株価追跡採点は行わない。
ただし、見送り判断も `trade_plans` に保存する。

理由：

* 買わなかった判断を後で検証できる
* 過剰取引を抑制する
* 「見送りの正しさ」を将来評価できる

### 10.3 役割

例：

* 決算前なので様子を見る
* 出来高がまだ足りない
* 追加開示を待つ

## 11. 取引計画ステータス遷移ルール

### 11.1 許可する状態遷移

| 現在          | 遷移先         | 条件              |
| ----------- | ----------- | --------------- |
| `DRAFT`     | `COMMITTED` | 計画種別ごとの必須条件を満たす |
| `COMMITTED` | `EXECUTED`  | 実行情報を入力する       |
| `COMMITTED` | `CLOSED`    | 実行せず終了する        |
| `EXECUTED`  | `CLOSED`    | 終了情報を入力する       |

---

### 11.2 禁止する状態遷移

| 現在          | 遷移先         | 理由           |
| ----------- | ----------- | ------------ |
| `COMMITTED` | `DRAFT`     | 確定後の後戻り防止    |
| `EXECUTED`  | `DRAFT`     | 実行済み計画の改ざん防止 |
| `CLOSED`    | `DRAFT`     | 終了済み計画の復活防止  |
| `CLOSED`    | `COMMITTED` | 終了済み計画の再利用防止 |

---

### 11.3 EXECUTED への遷移条件

`plan_status = EXECUTED` にする場合、以下を必須とする。

| 項目        | カラム                  |
| --------- | -------------------- |
| 実際エントリー価格 | `actual_entry_price` |
| 実際株数      | `actual_quantity`    |
| 実行日時      | `executed_at`        |

#### バリデーション

| ルール              | エラー条件                                 |
| ---------------- | ------------------------------------- |
| 実際エントリー価格は0より大きい | `actual_entry_price <= 0`             |
| 実際株数は0より大きい      | `actual_quantity <= 0`                |
| 実際株数は取引単位の倍数     | `actual_quantity % trading_unit != 0` |

v0.1では、`actual_quantity > recommended_quantity` でも記録は可能とする。
ただし、検証ログでサイズ超過として振り返る対象にする。

---

### 11.4 CLOSED への遷移条件

`plan_status = CLOSED` にする場合、`closed_at` を保存する。

`EXECUTED` の計画を `CLOSED` にする場合、v0.1では決済価格は必須にしない。
本格的な損益管理はv0.3以降で扱う。

---

## 12. 取引計画ロックルール

### 12.1 ロックタイミング

`plan_status` を `COMMITTED` に変更した時点で、取引計画をロックする。

---

### 12.2 ロック時に行う処理

| 項目                 | 処理                    |
| ------------------ | --------------------- |
| `is_locked`        | `true` にする            |
| `locked_at`        | 現在日時を保存する             |
| `locked_plan_hash` | ロック対象項目からハッシュ生成して保存する |

---

### 12.3 ロック対象項目

ロック後、以下の項目は更新不可とする。

## 13. 検証ログ作成ルール

### 13.1 対象テーブル

* `verification_logs`
* `trade_plans`

---

### 13.2 作成条件

検証ログは、必ず既存の `trade_plans.id` に紐づける。

`trade_plan_id` なしの検証ログは作成不可。

---

### 13.3 引き継ぎ項目

`trade_plans` から `verification_logs` を作成する場合、以下を引き継ぐ。

| verification_logs     | 引き継ぎ元                             |
| --------------------- | --------------------------------- |
| `trade_plan_id`       | `trade_plans.id`                  |
| `stock_code`          | `trade_plans.stock_code`          |
| `stock_name_snapshot` | `trade_plans.stock_name_snapshot` |
| `review_date`         | 初期値として当日                          |

---

### 13.4 followed_plan

`followed_plan` は以下を許容する。

| 値         | 内容     |
| --------- | ------ |
| `YES`     | 計画通り   |
| `NO`      | 計画から逸脱 |
| `PARTIAL` | 一部計画通り |
| `UNKNOWN` | 未評価    |

初期値は `UNKNOWN` とする。

---

### 13.5 deviation_summary の候補

v0.1では自由記述でよい。
ただし、画面上では候補選択を用意してもよい。

候補：

* 損切遅れ
* 損切未実行
* 早期利確
* 利確遅れ
* サイズ超過
* 最大保有日数超過
* 予定外エントリー
* 計画なしエントリー
* 監視のみで正解
* 見送りで正解
* 見送りで機会損失

## 14. 削除・終了ルール

### 14.1 基本方針

v0.1では、検証に関わるデータを安易に削除しない。

---

### 14.2 削除可能

| データ        | 条件                  |
| ---------- | ------------------- |
| 注目開示       | 関連する監視リスト・取引計画がない場合 |
| 監視リスト      | 関連する取引計画がない場合       |
| DRAFTの取引計画 | 未確定の場合              |
| キーワードルール   | 原則削除ではなく無効化推奨       |

---

### 14.3 削除不可

| データ              | 理由            |
| ---------------- | ------------- |
| COMMITTED以降の取引計画 | 事前計画の証跡を残すため  |
| EXECUTEDの取引計画    | 取引記録として残すため   |
| CLOSEDの取引計画      | 振り返り対象として残すため |
| 検証ログ             | 学習資産として残すため   |

---

### 14.4 終了扱い

削除しない代わりに、以下の状態にする。

| データ   | 終了方法                   |
| ----- | ---------------------- |
| 監視リスト | `status = CLOSED`      |
| 取引計画  | `plan_status = CLOSED` |

---

## 16. エラーメッセージ方針

### 16.1 基本方針

エラーは、原因と次に取るべき行動が分かる文言にする。

悪い例：

```text
入力エラーです。
```

良い例：

```text
損切価格はエントリー予定価格より低く入力してください。
```

---

### 16.2 主要エラー

| ケース       | メッセージ                              |
| --------- | ---------------------------------- |
| 銘柄コード不正   | 銘柄コードは4〜5文字の半角英数字で入力してください。        |
| 重複開示      | 同じ開示がすでに登録されています。                  |
| 仮説未入力     | 買い計画を確定するには、仮説を入力してください。           |
| 損切未入力     | 買い計画を確定するには、損切価格を入力してください。         |
| 利確未入力     | 買い計画を確定するには、利確目標価格を入力してください。       |
| 最大保有日数未入力 | 買い計画を確定するには、最大保有日数を入力してください。       |
| 損切価格不正    | 損切価格はエントリー予定価格より低く入力してください。        |
| 利確価格不正    | 利確目標価格はエントリー予定価格より高く入力してください。      |
| リスク%超過    | 許容リスク%が上限を超えています。                  |
| 推奨株数0     | この損切幅とリスク許容額では、最小単元を買うとリスク超過になります。 |
| 株数超過      | 予定株数が推奨株数を超えています。                  |
| 取引単位不一致   | 予定株数は取引単位の倍数で入力してください。             |
| ロック後編集    | この取引計画は確定済みのため、事前計画項目は編集できません。     |
| 検証ログ単独作成  | 検証ログは取引計画に紐づけて作成してください。            |

## 17. 実装責務分担

### 17.1 フロントエンドで行うこと

| 処理         | 理由              |
| ---------- | --------------- |
| 入力補助       | ユーザー体験向上        |

| 警告表示       | サイズ超過・推奨株数0を見せる |

---

### 17.2 サーバー側で必ず行うこと

以下はフロントで実施していても、必ずサーバー側で再検証する。

| 処理            | 理由         |
| ------------- | ---------- |
| 銘柄コード検証       | 不正データ防止    |
| 開示重複判定        | DB整合性維持    |
| キーワード分類       | 保存値の一貫性    |
| 買い計画確定バリデーション | 規律違反防止     |
| リスク逆算サイジング再計算 | 改ざん・計算ズレ防止 |
| サイズ超過ブロック     | リスク管理の中核   |
| 取引計画ロック       | 後知恵修正防止    |
| ロック後更新制御      | 証跡保全       |

---

### 17.3 DB制約で補助すること

| 処理             | 方法                          |
| -------------- | --------------------------- |
| 主キー            | CUID                        |
| 重複開示           | `unique_key` unique         |
| 銘柄コード一意        | `stocks.code` unique        |
| キーワード重複        | `keyword + category` unique |
| app_settings重複 | `key` unique                |
| 外部キー整合性        | Prisma relation / FK        |

複雑な条件付き必須やロック後更新不可は、v0.1ではアプリケーション層で実装する。

## 20. 最重要ルール

v0.1で最も重要なルールは以下である。

```text
出口未定の買い計画は確定できない。
```

```text
許容損失から逆算した推奨株数を超える買い計画は確定できない。
```

```text
確定後の事前計画は後から書き換えられない。
```

この3つが守られていない場合、本システムは単なるメモアプリになり、リスク管理ツールとして失敗である。

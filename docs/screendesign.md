## 2. 画面設計の基本方針

### 2.1 PCブラウザ優先

v0.1ではPCブラウザ利用を優先する。
スマートフォン最適化はv0.2以降で検討する。

### 2.2 毎日使えることを優先する

本アプリは、毎日5〜15分で使えることを重視する。

そのため、以下を優先する。

* 一覧で状況がすぐ分かる
* 入力項目を迷わない
* 注目開示から取引計画までの導線が短い
* 検証ログへ戻りやすい
* 危険な買い計画を画面上でも止める

### 2.3 取引計画画面を最重要画面にする

本アプリの中核は、銘柄発見ではなく、**出口とリスクを決めてから取引判断すること**である。

そのため、`/trade-plans` と `/trade-plans/[id]` を最重要画面とする。

### 2.4 UIライブラリ想定

v0.1では以下のUI部品を中心に使う。

| UI部品     | 用途                |
| -------- | ----------------- |
| Button   | 登録・更新・確定・実行       |
| Input    | 価格・株数・検索条件        |
| Textarea | 仮説・理由・学び          |
| Select   | 種別・ステータス・カテゴリ     |
| Dialog   | 登録フォーム            |
| Table    | 一覧表示              |
| Card     | サマリー・計算結果         |
| Badge    | カテゴリ・ステータス・ロック状態  |
| Alert    | 警告・エラー表示          |
| Tabs     | 詳細画面の切り替えに使用してもよい |

## 8.6 BUY / DRAFT のボタン制御

| ボタン       | 条件                   |
| --------- | -------------------- |
| 下書き保存     | 常に押下可能               |
| 買い計画を確定する | 必須項目・リスク条件を満たす場合のみ有効 |

### 確定ボタン有効条件

以下をすべて満たすこと。

* 仮説が入力されている
* エントリー予定価格が入力されている
* 損切価格が入力されている
* 利確目標価格が入力されている
* 最大保有日数が入力されている
* 口座資産が入力されている
* 許容リスク%が入力されている
* 推奨株数が1単元以上
* 予定株数が推奨株数以下
* 予定株数が取引単位の倍数

### 注意

フロントでボタン制御していても、API側で必ず再検証する。

## 8.7 BUY / DRAFT のエラー表示

# 15. 画面設計上の最重要ポイント

v0.1で最も重要な画面設計方針は以下である。

```text
BUY計画では、出口条件とリスク計算を入力しない限り、確定できないUIにする。
```

```text
COMMITTED後は、事前計画を編集フォームとして出さない。
```

```text
検証ログへ自然に進める導線を必ず置く。
```

この3点が実現できていない場合、本アプリはリスク管理ツールではなく、単なる銘柄メモアプリになってしまう。
